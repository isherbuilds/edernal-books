# Phase 00 Platform Foundation Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

> **Schema source of truth:** Follow `docs/superpowers/plans/2026-06-17-accounting-foundation-schema-revision-plan.md`. That amendment overrides older names and any copied reference-repo assumptions.

**Goal:** Establish organization-scoped platform foundation for the accounting app: auth, business settings, request context, RLS transaction helpers, audit events, outbox events, and idempotency ledger.

**Architecture:** Use Better Auth `organization` as the business tenant. Keep Better Auth-owned IDs and tables in the format generated by this repo; app-owned accounting/settings tables reference those IDs using matching column types. Shared contracts live in `packages/core`, persistence lives in `packages/db`, API wiring lives in `packages/api`, auth config lives in `packages/auth`, and UI lives in `apps/web`.

**Tech Stack:** TanStack Start, Hono, oRPC, Drizzle, PostgreSQL, Better Auth, Zod, Vite Plus, Vitest.

Plain-language note:

- `withOrgSnapshotRead` means "read data through RLS using a stable database transaction for reports/exports." It does not create an organization snapshot table.

- [ ] **Step 2: Implement table classification**

```ts
const betterAuthTables = new Set([
  "user",
  "session",
  "account",
  "verification",
  "organization",
  "member",
  "invitation",
  "api_key"
]);
const globalTables = new Set(["currency"]);

export type TableInventory = {
  name: string;
  columns: string[];
};

/**
 * Classifies a database table for tenant safety.
 *
 * Why this exists: every app-owned business table must either be protected by
 * `organization_id` + RLS, explicitly global, or explicitly owned by Better Auth.
 * This catches accidental unscoped tables before they ship.
 */
export function classifyTableForRls(table: TableInventory) {
  if (betterAuthTables.has(table.name)) {
    return { kind: "auth-owned" as const, reason: "Better Auth table" };
  }

  if (globalTables.has(table.name)) {
    return { kind: "global" as const, reason: "global reference table" };
  }

  if (table.columns.includes("organization_id")) {
    return { kind: "tenant" as const, reason: "has organization_id" };
  }

  return { kind: "invalid" as const, reason: "app-owned table missing organization_id" };
}
```

- [ ] **Step 3: Add transaction helpers**

`packages/db/src/tx.ts` sets the organization context inside every tenant transaction:

```ts
import { sql } from "drizzle-orm";
import { db } from "./index";

/**
 * Runs tenant-scoped work inside one database transaction.
 *
 * Why this exists: PostgreSQL RLS policies read `app.current_org_id`, so every
 * tenant read/write must set it before touching app-owned tables.
 */
export async function withOrgTx<T>(
  organizationId: string,
  run: (tx: Parameters<Parameters<typeof db.transaction>[0]>[0]) => Promise<T>
) {
  return db.transaction(async (tx) => {
    await tx.execute(sql`select set_config('app.current_org_id', ${organizationId}, true)`);
    return run(tx);
  });
}

/**
 * Read wrapper for tenant-scoped queries.
 *
 * Today it delegates to `withOrgTx`; keeping a separate name lets services show
 * intent and allows read-specific transaction behavior later.
 */
export async function withOrgRead<T>(
  organizationId: string,
  run: (tx: Parameters<Parameters<typeof db.transaction>[0]>[0]) => Promise<T>
) {
  return withOrgTx(organizationId, run);
}

/**
 * Read wrapper for reports/exports that need a consistent view of rows.
 *
 * "Snapshot" means a stable database view during the read, not product inventory
 * and not an `organization` data snapshot table.
 */
export async function withOrgSnapshotRead<T>(
  organizationId: string,
  run: (tx: Parameters<Parameters<typeof db.transaction>[0]>[0]) => Promise<T>
) {
  return withOrgTx(organizationId, run);
}
```

export type AppContext = {
  userId: string;
  organizationId?: string;
  role?: string;
  requestId: string;
  ip?: string;
  userAgent?: string;
};

/**
 * Returns the active business ID or fails before service code touches tenant data.
 *
 * Why this exists: API handlers should not rely on optional organization context
 * when reading or writing app-owned tables protected by RLS.
 */
export function requireOrganization(context: AppContext) {
  if (!context.organizationId) {
    throw new Error("ORGANIZATION_REQUIRED");
  }

  return context.organizationId;
}

```ts
export function buildOrganizationSettingEvents(input: {
  organizationId: string;
  userId: string;
  requestId: string;
  legalName: string;
}) {
  // Creates the side-effect records that make settings changes auditable and
  // discoverable by background processors without duplicating business logic.
  return {
    audit: {
      organizationId: input.organizationId,
      userId: input.userId,
      entityType: "organization_setting",
      entityId: input.organizationId,
      action: "organization_setting.upserted",
      payload: {
        before: null,
        after: { legalName: input.legalName },
        metadata: { requestId: input.requestId, source: "web" }
      }
    },
    outbox: {
      organizationId: input.organizationId,
      aggregateType: "organization_setting",
      aggregateId: input.organizationId,
      eventType: "organization_setting.upserted",
      payload: { legalName: input.legalName }
    }
  };
}
```

## Phase 0 Completion Gate

Do not start Phase 1 until:

- Better Auth signup/login and organization membership work.
- One business has one `organization_setting` row.
- Role checks are server-side.
- `audit_event`, `outbox_event`, and `idempotency_ledger` exist.
- Sensitive settings mutation writes `audit_event` and `outbox_event` in the same transaction.
- App-owned tenant tables have `organization_id`, RLS enabled, and matching policies.
- Better Auth-owned tables are excluded from app RLS checks.
- Runtime role cannot bypass RLS before production deployment.
- No Phase 1/2/3 tables are exported from `schema/migration.ts`.
- Commands use `vp`/root scripts and imports use `@tsu-stack/*`.
